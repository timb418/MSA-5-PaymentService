# ADR-001: Размещение функции Saga-оркестрации

## Общая информация

1. Родительский артефакт — архитектура платёжной платформы OrchestrPay
2. Автор ADR — Тимур Б.
3. Статус — Принят
4. Согласующие — Tech Lead, команда «Безопасность», команда DevOps/SRE

## Контекст

Платёжный процесс OrchestrPay включает несколько микросервисов: **Payment Service** (списание и перевод средств), **FraudCheck Service** (антифрод-проверка с тремя исходами: разрешение / запрет / ручная проверка) и **Notification Service** (уведомления клиента). В рамках одной транзакции задействованы все три сервиса, а при сбое на любом шаге необходимо выполнить компенсационные операции (возврат средств, откат статусов).

Для реализации Saga-оркестрации необходимо принять решение: **где разместить логику управления Saga** — внутри существующего доменного сервиса (Payment Service) или в выделенном сервисе-оркестраторе.

Ключевые ограничения и требования, влияющие на решение:
- Нельзя допустить ситуации, когда деньги зависают между системами.
- Компенсационные операции должны выполняться автоматически.
- Требуется поддержка состояния ожидания при ручной антифрод-проверке (до 20 минут).
- Необходим прозрачный мониторинг хода транзакции для бизнеса и службы безопасности.
- SLA: менее 5 секунд для 90% успешных транзакций.
- В качестве BPM-движка выбрана Camunda (Zeebe).

## Рассмотренные варианты

### Вариант №1 «Оркестратор внутри Payment Service»

Логика Saga-оркестрации встраивается непосредственно в Payment Service. Этот сервис берёт на себя ответственность за последовательный вызов FraudCheck Service и Notification Service, отслеживание состояния и запуск компенсаций.

```
Client → Payment Service ──► FraudCheck Service
              │
              ▼ (оркестрирует)
         Notification Service
```

Payment Service содержит:
- BPMN-процесс (Camunda Zeebe worker)
- Логику управления состоянием Saga
- Обработчики компенсационных операций
- Доменную логику списания/перевода средств

### Вариант №2 «Отдельный Orchestrator Service»

Логика Saga-оркестрации выносится в независимый **Orchestrator Service**, который управляет процессом через Camunda Zeebe. Доменные сервисы реализуют только свои Job Workers и остаются без знания об общем процессе.

```
Client → Orchestrator Service (Camunda Zeebe)
              │
              ├──► Payment Service  (worker: DEBIT_ACCOUNT, TRANSFER_FUNDS)
              ├──► FraudCheck Service (worker: FRAUD_CHECK)
              └──► Notification Service (worker: SEND_NOTIFICATION)
```

Orchestrator Service содержит:
- BPMN-процесс и его деплой в Zeebe
- Job Workers только для координационной логики
- Управление состоянием Saga и таймаутами

Каждый доменный сервис содержит только:
- Свои Job Workers с доменной логикой
- Собственные данные и ответственность

## Сравнение альтернатив

| Вариант решения | Преимущества | Недостатки |
|---|---|---|
| Вариант №1 «Оркестратор внутри Payment Service» | - Меньше сетевых вызовов при старте<br>- Проще первоначальный деплой<br>- Один сервис для отладки на ранних этапах | - Нарушение принципа единой ответственности (SRP)<br>- Payment Service становится God Object<br>- Сложность независимого масштабирования (оркестратор и домен растут вместе)<br>- Изменения в процессе оркестрации требуют деплоя Payment Service<br>- Высокая связность: сбой оркестратора = сбой домена<br>- Сложнее обеспечить изоляцию состояния Saga от доменных данных |
| Вариант №2 «Отдельный Orchestrator Service» | - Соблюдение SRP: каждый сервис отвечает только за свой домен<br>- Независимое масштабирование оркестратора и доменных сервисов<br>- Изменения бизнес-процесса не затрагивают доменные сервисы<br>- Прозрачная визуализация процесса через Camunda Operate<br>- Отказ одного доменного сервиса не ломает оркестратор<br>- Лёгкое добавление новых шагов без изменения существующих сервисов<br>- Явная точка входа для мониторинга и трассировки всей Saga | - Дополнительный сервис для разработки, деплоя и поддержки<br>- Дополнительные сетевые вызовы через Zeebe брокер<br>- Необходимо настроить Camunda инфраструктуру (Zeebe, Operate) |

## Принятое решение

**Вариант №2 — Отдельный Orchestrator Service.**

Платёжный процесс OrchestrPay по своей природе является распределённой транзакцией с компенсационной логикой, состоянием ожидания и участием трёх независимых доменных сервисов. Встраивание оркестрации в Payment Service создаст тесную связность и сделает его ответственным за чужую доменную логику, что недопустимо для производственной финансовой системы.

Выделенный Orchestrator Service обеспечивает:
- **Надёжность**: изоляция отказов между слоем оркестрации и доменным слоем.
- **Наблюдаемость**: Camunda Operate даёт полную картину состояния каждой транзакции в реальном времени — критически важно для службы безопасности и бизнеса.
- **Масштабируемость**: оркестратор и доменные сервисы масштабируются независимо в соответствии с профилем нагрузки.
- **Сопровождаемость**: изменения бизнес-процесса (новые шаги проверок, изменение таймаутов) деплоятся только в Orchestrator Service, не затрагивая доменные сервисы.
- **Соответствие требованиям**: cut-off таймер, ожидание ручной проверки до 20 минут и автоматические компенсации естественно выражаются в BPMN-нотации Camunda Zeebe.

# Task 6. Fitness-функции для платёжной системы OrchestrPay

## Таблица fitness-функций

| Название функции | Назначение | Что проверяется | Действие при нарушении | Способ реализации |
|---|---|---|---|---|
| **— Консистентность данных между микросервисами —** | | | | |
| FundsBalanceConsistency | Консистентность сумм между Payment Service и Transfer Service | Сумма, списанная при DEBIT_ACCOUNT, совпадает с суммой, зачисленной при CREDIT_ACCOUNT контрагенту | Заморозка транзакции, алерт команде финансов, создание инцидента для ручного разбора | Scheduled job (каждые 5 мин): сверка записей в таблицах `debit_log` и `transfer_log` по `payment_id`; расхождение > 0 считается нарушением |
| NotificationDeliveryConsistency | Гарантия доставки уведомлений клиенту | Для каждого платежа в статусе COMPLETED, FAILED или REFUNDED существует запись об успешной отправке уведомления в таблице `notification_log` | Повторная отправка уведомления через Notification Service; если 3 попытки неуспешны — алерт Customer Support | Scheduled job (каждые 2 мин): LEFT JOIN `payments` и `notification_log` по `payment_id`; строки без записи уведомления — нарушение |
| **— Зависшие процессы —** | | | | |
| HungProcessDetector | Обнаружение платежей, застрявших в промежуточном состоянии | Платежи в состояниях FUNDS_RESERVED, FRAUD_CHECK_IN_PROGRESS или TRANSFER_IN_PROGRESS дольше 10 минут | Инициирование компенсационной Saga (откат списания), перевод в статус FAILED, алерт DevOps/SRE | Cron job (каждые 5 мин): `SELECT * FROM payments WHERE status IN ('FUNDS_RESERVED', 'FRAUD_CHECK_IN_PROGRESS', 'TRANSFER_IN_PROGRESS') AND updated_at < NOW() - INTERVAL '10 minutes'` |
| CamundaJobRetryExhaustion | Обнаружение Job Workers с исчерпанными попытками повтора | Job-инстансы в Camunda с `retries = 0` (все retry использованы, процесс остановлен) | Перемещение в Dead Letter Queue, уведомление DevOps-команды, создание инцидента в системе алертинга | Scheduled job (каждые 1 мин): запрос к Camunda REST API `/v1/incidents?type=FAIL&state=ACTIVE`; анализ jobs с `retries=0` |
| **— Целостность Saga-транзакций —** | | | | |
| SagaStateConsistency | Консистентность состояний Saga | У всех Saga есть корректные конечные состояния (COMPLETED, FAILED, REFUNDED); отсутствуют записи, «застрявшие» в незавершённых состояниях без активного BPMN-процесса | Восстановление состояния из лога событий или откат до предыдущего стабильного состояния | Scheduled job (каждые 10 мин): сверка таблицы `saga_states` с активными инстансами в Camunda; несоответствие финального статуса — нарушение |
| CompensationIntegrityCheck | Гарантия запуска компенсации при неудаче после pivot-точки | Для каждого FAILED-платежа, у которого был выполнен шаг TRANSFER_FUNDS (pivot), существует запись компенсационной транзакции REFUND в таблице `compensation_log` | Принудительный запуск компенсационного процесса через Camunda Message Correlation; алерт команде финансов | Scheduled job (каждые 5 мин): `SELECT * FROM payments p LEFT JOIN compensation_log c ON p.id = c.payment_id WHERE p.status = 'FAILED' AND p.pivot_reached = true AND c.id IS NULL` |
| **— Таймауты и дедлайны —** | | | | |
| PaymentTimeoutControl | Контроль максимального времени выполнения платежа | Платежи в статусе PROCESSING дольше 30 минут | Автоматический возврат средств клиенту, перевод платежа в статус REFUNDED, уведомление клиента через Notification Service | Cron job (каждые 5 мин): `SELECT * FROM payments WHERE status = 'PROCESSING' AND created_at < NOW() - INTERVAL '30 minutes'`; инициирование refund-процесса |
| ManualReviewDeadlineMonitor | Контроль дедлайна ручной антифрод-проверки (cut-off-time) | Платежи в статусе MANUAL_REVIEW_PENDING дольше 20 минут без принятия решения оператором | Автоматическое одобрение транзакции (согласно бизнес-правилу cut-off), отправка сигнала в Camunda для продолжения процесса, уведомление команды безопасности | Cron job (каждые 1 мин): `SELECT * FROM payments WHERE status = 'MANUAL_REVIEW_PENDING' AND review_started_at < NOW() - INTERVAL '20 minutes'`; отправка Camunda Message `manual_review_approved` |
| FraudCheckResponseTimeout | Контроль времени ответа FraudCheck Service | Запросы к FraudCheck Service, не получившие ответ в течение 5 секунд (SLA системы — менее 5 сек в 90% случаев) | Retry с exponential backoff (3 попытки); при исчерпании retry — автоматическая эскалация на ручную проверку; алерт DevOps | Метрика в Prometheus: `histogram_quantile(0.90, fraud_check_response_duration_seconds)`; Alert rule при p90 > 5s; дополнительно — dead letter анализ Camunda для job-типа `fraud-check` |
